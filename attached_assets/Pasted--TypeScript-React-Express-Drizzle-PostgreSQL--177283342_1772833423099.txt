あなたはTypeScript + React + Express + Drizzle(PostgreSQL)構成の既存プロジェクトを、安全に大規模リファクタリングするシニアエンジニアです。
目的は「Notion依存を撤廃し、アプリ内DBのみで完結する構成」に移行しつつ、UI/機能をシンプル化することです。

## ゴール
1) データベース単一化
- Notion API連携を全廃し、PostgreSQL（Drizzle）だけで完結させる。
- 既存のNotion同期/投稿処理（API・ロジック・環境変数依存）を削除。
- 申請データはすべてアプリ内DBへ保存し、管理者承認フローをDBのstatus等で扱える設計にする。

2) ホーム画面削除 + イベント画面をメイン化
- "/" をイベント一覧ページに変更（旧homeページは削除）。
- イベント一覧上部に「軽いホーム感（ファーストビュー）」を追加。
  例: サイトの価値提案、簡単な導線、視覚的ヒーロー要素。
- ただし過度に情報量を増やさず、イベント探索を主役にする。

3) 不要機能の削除
- 「仲間を探す」機能を削除（UI導線・ルート・関連API・不要schemaを整理）。
- 「記事」タブを削除（UI導線・ルート・関連API・不要schemaを整理）。
- 削除後に未使用import/type/endpointが残らないようにする。

4) お問い合わせ導線の再設計
- ナビ表記を「お問い合わせ＆掲載依頼」に変更。
- 掲載依頼は以下を1画面で扱う:
  - 必須: 団体紹介情報
  - 任意: イベント情報
- 「Googleフォーム風」ではなく、実際の公開ページの見栄え（カード/プレビュー）で入力できるUIにする。
  - 入力中に「掲載後の見え方プレビュー」がリアルタイムで見えること
  - できるだけ本番表示コンポーネントを再利用し、二重管理を減らすこと

5) メール通知
- 管理者メールアドレスを環境変数で設定し、掲載依頼送信時に管理者へ通知メール送信。
- 掲載依頼者にメールアドレス入力を必須化。
- 掲載確定時に掲載依頼者へ通知メールを送信できるようにする（まずはサーバー側APIを実装）。
- メール送信失敗時のリトライ/ログ方針を最低限実装（fail-safe）。

## 既存構成を踏まえた実装方針（必須）
- バックエンド: Express routes, server/storage.ts, shared/schema.ts を中心に変更。
- フロント: wouterルーティング（App.tsx）・headerナビ・pages配下を整理。
- Drizzle schema変更後は必要なmigration/`db:push`想定の整合性を取る。
- 型安全を維持し、Zodバリデーションを適切に更新。
- UI文言は日本語の自然さを優先。

## 具体タスク
A. Notion依存除去
- server/notion.ts, server/notion-sync.ts, notion関連routeを削除または無効化。
- package.json から Notion依存パッケージ削除。
- .env想定からNotion系変数を不要化（README or docsに反映）。

B. ルーティング/ナビ改修
- "/" => EventsPage
- home.tsx削除（または未使用化して完全除去）
- buddies/articles系route削除
- headerのnav itemから「仲間を探す」「記事」を除去
- 「お問い合わせ」=>「お問い合わせ＆掲載依頼」へ変更

C. お問い合わせ＆掲載依頼の新データ設計
- contact_submissionsを拡張 or 新規submissionテーブルを作成し以下を保持:
  - requesterEmail(必須), requesterName(任意), 団体情報(必須), イベント情報(任意), status, createdAt, updatedAt
- 「団体のみ申請」「団体+イベント申請」を両対応
- 入力値は公開表示と同じ構造を意識したschemaにする（後で承認時にそのまま反映しやすい形）

D. メール実装
- メール送信サービス層を作成（例: server/services/mail.ts）
- 管理者通知: 新規掲載依頼時
- 申請者通知: 掲載確定時
- providerはSMTPまたは外部APIを抽象化し、環境変数未設定時は安全にスキップ+警告ログ
- API:
  - POST /api/submissions (新規申請)
  - POST /api/submissions/:id/approve (承認 + 掲載確定メール)
  - 必要なら簡易admin認証を実装

E. UI/UX
- Contactページを「お問い合わせ＆掲載依頼」に統合
- 掲載依頼フォームは公開カード風UIで入力（preview付き）
- 団体情報セクションを先頭・必須、イベント情報は折りたたみ可能で任意
- 送信完了/失敗トーストを明確化

F. クリーンアップ
- 削除した機能に紐づく未使用型・API・テーブルを整理
- tscエラーゼロ、lint想定エラーゼロ

## 受け入れ条件（Acceptance Criteria）
- Notion関連コードと依存が実質的に残っていない
- "/" でイベント画面が開く
- ナビに「仲間を探す」「記事」が存在しない
- 「お問い合わせ＆掲載依頼」画面で、団体必須・イベント任意の申請ができる
- 掲載依頼者メールが必須
- 新規申請時に管理者通知メール処理が呼ばれる
- 承認APIで申請者への掲載確定メール処理が呼ばれる
- 型チェック（`npm run check`）が通る
